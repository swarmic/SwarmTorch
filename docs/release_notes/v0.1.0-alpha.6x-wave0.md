# SwarmTorch v0.1.0-alpha.6x — Wave 0 Remediation

**Date:** 2026-02-24  
**Commit:** `1388912` (pushed to `main`)  
**Type:** Security/correctness hotfixes — replay, identity, fingerprint, clock, telemetry  
**Builds on:** `v0.1.0-alpha.6x` (HEAD `ad75fff` at audit baseline)  
**Source:** [Deep Codebase Audit Report](../plans/v0.1.0-alpha.6x-remediation-id-map.md), IDs C-01 through C-03, H-01 through H-03

## Summary

Wave 0 closes the six highest-priority findings from the deep codebase audit. All six are security-relevant or correctness-critical and were identified as mandatory-before-release.

| ID | Severity | Slice | One-liner |
|---|---|---|---|
| C-01 | Critical | R1 | Replay boundary bypass — sequences at exact tolerance edge could be replayed |
| C-02 | Critical | R2 | Fail-open hashing — serialization errors silently produced sentinel hashes |
| C-03 | Critical | R3 | Embassy clock returns zero — falsifies all timestamps in protocol layer |
| H-01 | High | R1 | PeerId identity divergence across `std`/`no_std` profiles |
| H-02 | High | R1 | Replay cache looked up by raw sender bytes, not canonical PeerId |
| H-03 | High | R3 | All-zero RunId sentinel causes telemetry collision across all trait-fallback calls |

## Key Changes

### R1 — Replay/Auth Identity Contract Hardening

#### C-01: Replay Boundary Bypass

**File:** `swarm-torch-core/src/replay.rs`

Changed `validate_and_update` boundary comparison from strict `<` to `<=`, closing a window where sequences exactly at the tolerance edge passed validation but were then pruned, allowing replay on resubmission.

```diff
-if max_acceptable_oldness < self.last_sequence {
+if max_acceptable_oldness <= self.last_sequence {
```

#### H-01: PeerId Canonicalization

**Files:** `swarm-torch-core/src/traits.rs`, `swarm-torch-core/src/crypto.rs`

- Removed `#[cfg(feature = "std")]` gate from `PeerId::from_public_key()` — SHA-256 derivation is now available unconditionally.
- Unified `KeyPair::peer_id()` to always use canonical hash derivation. Previously, `no_std` fell back to raw byte identity, causing cross-profile identity mismatches.

#### H-02: Verify Path Identity Consistency

**Files:** `swarm-torch-net/src/protocol.rs`

- Changed `verify_authenticated()` replay cache lookup from `PeerId::new(self.sender)` (raw bytes) to `PeerId::from_public_key(&self.sender)` (canonical hash).
- Ungated `sender_peer_id()` — now available in all profiles.
- Ungated `PeerId` import to match `sender_peer_id()` visibility (fixes `--no-default-features` regression caught in post-commit review).

---

### R2 — Fingerprint/Source Descriptor Hardening

#### C-02: Fail-Closed Hashing

**Files:** `swarm-torch-core/src/dataops.rs`, `swarm-torch-core/src/run_graph.rs`, `swarm-torch/src/artifacts.rs`, `swarm-torch/src/report.rs`

Two hashing functions previously swallowed serialization errors and returned sentinel all-zero hashes:

- `sha256_postcard()` in `dataops.rs`
- `node_def_hash_v1()` in `run_graph.rs`

Both now return `Result<[u8; 32], postcard::Error>`. This change propagates through:

| Layer | Functions updated |
|---|---|
| `dataops.rs` | `source_fingerprint_v0`, `schema_hash_v0`, `recipe_hash_v0`, `dataset_fingerprint_v0`, `no_schema_hash_v0`, `derived_source_fingerprint_v0`, `dataset_entry_v1`, `derived_dataset_entry_v1`, `cache_key_v0`, `predict_output_fingerprints` |
| `run_graph.rs` | `normalize_node_v1`, `GraphV1::normalize` |
| `artifacts.rs` | `register_source`, `materialize_node_outputs`, `predict`, `write_graph` |
| `report.rs` | `load_report` |

All production callers map `postcard::Error` to `io::Error(InvalidData)` at IO boundaries. Test callers use `.unwrap()`.

---

### R3 — Runtime Clock + Telemetry Artifact Integrity

#### C-03: Embassy Clock Fail-Fast

**File:** `swarm-torch-runtime/src/lib.rs`

Replaced `EmbassyRuntime::now()` zero-return stub with `unimplemented!()`. Returning zero silently falsified every timestamp flowing through the protocol layer. The fail-fast gate prevents accidental use until a real `embassy_time::Instant` driver is wired.

#### H-03: Deterministic RunId Derivation

**File:** `swarm-torch/src/native_runner.rs`

Replaced `RunId::from_bytes([0u8; 16])` all-zero sentinel with a per-invocation deterministic id: `SHA-256("swarmtorch.trait_fallback_run_id" || clock_nanos)[..16]`. This prevents telemetry from different trait-level calls silently colliding under a shared zero id.

- `unused import: alloc::vec::Vec` in `swarm-torch-net/src/traits.rs:9` under alloc-only builds — `Vec` is used by `SwarmTransport` which is `std`-gated, not `alloc`-gated. Tracked as L-02 in the remediation backlog.

## Compatibility Notes

- **Breaking:** `sha256_postcard()`, `node_def_hash_v1()`, `GraphV1::normalize()`, and 10 `dataops` functions now return `Result`. Callers must handle the error.
- **Non-breaking:** `PeerId::from_public_key()` is now available in all profiles (was `std`-only).
- **Non-breaking:** `sender_peer_id()` is now available in all profiles (was `std`-only).
- **Behavioral:** `EmbassyRuntime::now()` will panic at runtime if called (was silent zero). Only affects the experimental embassy feature flag.
- **Behavioral:** Trait-fallback `OpRunner::run` now produces unique RunIds per call (was all-zero).
