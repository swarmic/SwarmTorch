# v0.1.0-alpha.8x Phase 2 Remediation Roadmap

Status: Active (post-Wave 8.0 follow-on)  
Date: 2026-04-05  
Scope: Security/runtime/integrity follow-ons intentionally deferred from Wave 8.0 implementation lock

## Purpose

This roadmap tracks validated follow-up work that should not be bundled into a single implementation wave.
Partitioning is based on blast radius, trust-boundary impact, ADR relevance, and release risk.

## Phase Partitioning Decision (Approved)

### Wave 8.1 (Immediate Next Phase: low-risk / high-leverage)

1. CI supply-chain enforcement (`cargo deny`, `cargo audit`, `cargo vet`).
2. Dedicated Windows CI lane validating symlink-escape behavior in artifact manifest checks.
3. Telemetry key migration follow-through for downstream artifact consumers (`swarmtorch.*` keys).

Rationale:
- Tightens assurance quickly with low core-runtime coupling.
- Validates cross-platform trust-boundary behavior.
- Closes migration friction from Wave 8 telemetry key rename.

### Wave 8.2 (Security Semantics Phase)

1. `SecurityConfig` enforcement wiring (or controlled deprecation if ADR decision rejects wiring).
2. Persistent replay state backend to close restart/LRU replay windows.
3. Verifier membership/authorization hook on envelope acceptance boundary.
4. Cross-process artifact writer safety policy implementation.

Rationale:
- These are trust-boundary and security semantics changes and should land together under one security-focused validation campaign.
- ADR-0008/0008B alignment is central; avoid scattering auth/replay semantics across unrelated waves.

### Wave 8.3 (Artifact Authenticity / Longer-Horizon)

1. Signed authenticity/provenance for artifact bundles/manifests (beyond hash-integrity consistency).
2. Benchmark harness modernization for convergence/topology/staleness release gates.
3. Embassy runtime `now()` placeholder replacement.

Rationale:
- Artifact signing/provenance requires key-management and contract decisions beyond Wave 8.1/8.2 scope.
- Performance/runtime uplift items are lower urgency and should not block security closure.

1. Every item in the matrix is closed with merged code/docs evidence.
2. Security-sensitive items (`P2-01`, `P2-04`, `P2-05`, `P2-10`) are CI or integration-gated.
3. Cross-platform artifact trust-boundary behavior is validated (`P2-08`).
4. `SECURITY.md`, whitepaper, and release notes match landed enforcement/provenance reality with no stale claims.
