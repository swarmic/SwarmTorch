# SwarmTorch v0.1.0-alpha.6x — Wave 1 Release Notes

**Date:** 2026-02-26  
**Type:** Safety Guards + Descriptor/Clock Hardening  
**Baseline:** Post-Wave 0 commit `1388912`

## Summary

Wave 1 adds **safety bounds** to aggregation, compression, and model construction, **hardens fingerprint derivation** by stripping URI noise, and **replaces the non-monotonic system clock** with a process-local monotonic source. All changes are fail-closed.

---

## Key Changes

### R4 — Aggregation / Compression / Model Safety

**H-04: Krum max-peers guard** (`aggregation.rs`)  
Added `MAX_KRUM_PEERS = 50`. `Krum::aggregate()` now rejects `n > 50` peers with `Error::ResourceExhausted` before the O(n²) distance matrix allocation.

**H-08: Gradient dimension bound** (`aggregation.rs`)  
Added `MAX_GRADIENT_DIM = 10_000_000`. `validate_gradient_shapes()` rejects oversized dimensions with `Error::InvalidGradient` before any aggregator allocates.

**H-05: Checked index casting** (`compression.rs`)  
Replaced lossy `*i as u32` with `try_usize_to_u32()` helper returning `CompressionError::IndexOverflow`. `compress()` now returns `Result<Self, CompressionError>`.

**H-06: Fail-closed decompression** (`compression.rs`)  
`decompress()` now returns `Result<Vec<f32>, CompressionError>` with three validation paths:
- Dense: byte length must be `% 4 == 0` and match `num_elements`
- Sparse: `values.len() / 4 == indices.len()` and all indices in-bounds
- Unmatched method/data combinations: `Err(CompressionError::InvalidData)` (previously returned zeroed gradients)

**H-07: Overflow-safe model construction** (`simple.rs`)  
`LinearModel::new()` and `SimpleMLP::new()` now return `Result<Self, ModelError>`. Dimension validation uses `checked_mul` to prevent silent wrapping on extreme inputs.

---

### R2 — Fingerprint Hardening

**M-01: URI query/fragment stripping** (`dataops.rs`)  
`normalize_and_redact_source_descriptor()` now strips `?...` query and `#...` fragment from URIs before fingerprinting. Semantically equivalent URIs with differing query parameters now produce identical fingerprints.

---

### R3 — Runtime Clock Hardening

**M-06: Monotonic clock** (`swarm-torch-runtime/src/lib.rs`)

> [!WARNING]
> **Behavioral contract change**: `SwarmRuntime::now()` on the Tokio runtime now returns **process-local monotonic milliseconds** (via `OnceLock<Instant>`), not UNIX epoch wall-clock milliseconds. Consumers interpreting the value as absolute time must use `SystemTime` directly.

Rationale: `SystemTime` can go backward on NTP step-backs, producing non-monotonic timestamps that violate protocol assumptions. `Instant` is guaranteed monotonic by the Rust standard library.

The trait-level `SwarmRuntime::now()` doc-comment (`lib.rs:26`) was updated to specify the monotonic contract:
- The returned value is monotonically non-decreasing milliseconds
- The epoch is implementation-defined (not necessarily UNIX epoch)
- Callers must not interpret this value as wall-clock time
- Implementations must guarantee the value never decreases between calls

## Behavioral Changes

| Area | Before (alpha.6) | After (Wave 1) |
|------|-------------------|-----------------|
| Sparse OOB indices | Silently ignored | `Err(CompressionError::IndexOutOfBounds)` |
| Unmatched method/data | Returns zeroed gradients | `Err(CompressionError::InvalidData)` |
| `compress()` | Infallible | `Result<Self, CompressionError>` |
| `decompress()` | Infallible | `Result<Vec<f32>, CompressionError>` |
| `LinearModel::new()` | Panics on invalid dims | `Result<Self, ModelError>` |
| `SimpleMLP::new()` | Panics on invalid dims | `Result<Self, ModelError>` |
| `TokioRuntime::now()` | Wall-clock ms (UNIX epoch) | Monotonic ms (process-local epoch) |
| `SwarmRuntime::now()` trait doc | Generic "Get the current instant" | Monotonic contract with implementation-defined epoch |
